## Why This Exists

Most practitioner websites are a page builder subscription, a booking plugin, and a theme that looks like everybody else. This one is a real site you own: your name, your prices, your booking link, your writing, and free readings that actually calculate something.

The readings are the difference. A visitor arrives, draws their own birth chart or pulls three cards, gets a real result, and lands on a booking button while it is still on the screen. Every reading is computed by [RoxyAPI](https://roxyapi.com) and verified against NASA JPL Horizons, not generated by a language model and hoped for.

## Quickstart

**1. Get the site running**

```bash
git clone https://github.com/RoxyAPI/spiritual-practitioner-website-template.git
cd spiritual-practitioner-website-template
npm install
npm run dev
```

**2. Add your key.** Get one at [roxyapi.com/pricing](https://roxyapi.com/pricing), then copy `env.example` to `.env.local`:

```bash
ROXYAPI_KEY=your_key_here
```

One key covers every reading on the site. Without it, the site still builds and every page still works; the readings simply say they are not connected yet.

**3. Make it yours.** Open `src/config/site.config.ts` and replace the example practitioner with yourself: name, bio, services, prices, testimonials, FAQ, booking link, palette. Then drop your own photo over `public/portrait.jpg` (keep it 1200x1600, a 3:4 portrait).

That is the whole customisation. No component needs editing for a standard fork.

## The Free Readings

Each reading is calculated from the visitor birth details on your server, using your key, and rendered with a drop in component. Nothing is stored.

| Reading | What the visitor gives | What they get |
|---|---|---|
| Birth chart | Date, time, city | The full natal wheel: planets, houses, aspects |
| Daily horoscope | Sun sign | A reading for today, written fresh each day |
| Compatibility | Two sets of birth details | A score with the category breakdown behind it |
| Life path number | Date | The number and what it asks of them |
| Three card tarot | An optional question | Past, present, and direction, with the reading |
| Human design | Date, time, city | Type, strategy, authority, then the full bodygraph |

Turn any of them off in the config and the page returns 404 and the link disappears from the navigation, the footer, the home page, and the sitemap. Nothing to clean up by hand.

## Booking and Contact

| | Cal.com | Calendly |
|---|---|---|
| Embed on your site | Yes, free | Yes, free |
| Match the embed to your palette | Yes | Paid plans |
| Take payment at booking | Yes, free | Paid plans |
| Pricing | [cal.com/pricing](https://cal.com/pricing) | [calendly.com/pricing](https://calendly.com/pricing) |

Set `booking.provider` and `booking.url` in the config. Until you do, the booking page shows a short setup card instead of a broken embed.

The contact form posts straight to [Web3Forms](https://web3forms.com), which mails it to you. The access key is publishable by design, so it lives in the config rather than in an environment variable, which is why deploying needs exactly one secret. Free tier and alternatives (Formspree, Basin) are in [`docs/integrations.md`](docs/integrations.md).

Payments stay yours: give any service a `paymentLink` (your own Stripe, PayPal, or similar page) and it appears on the services page. The site never touches money.

## Writing

Posts are files. Copy any file in `content/blog/`, rename it (the filename becomes the URL), edit the frontmatter, write the post, commit. Three posts ship with the template so the blog is not empty on day one. Replace them.

## Palettes

`rosewater` (default), `eucalyptus`, `kiln`, `moonlit`. Each ships a light and a dark variant, contrast checked to WCAG AA or better. Change one word in the config:

```ts
palette: 'moonlit',
```

The colours carry through every page, the reading components, and the social card. Full token tables: [`docs/design.md`](docs/design.md).

## Privacy

- Bookings live with your booking provider, under your account.
- Contact messages go to your inbox. Nothing is stored on the site.
- The readings send birth details to the calculation API at the moment they are requested. The site keeps none of it. There is no database to keep it in.

## Stack

| Layer | Choice |
|---|---|
| Framework | Next.js 16, App Router, Server Components |
| Styling | Tailwind CSS v4 and shadcn/ui |
| Readings | [`@roxyapi/sdk`](https://www.npmjs.com/package/@roxyapi/sdk) on the server, [`@roxyapi/ui-react`](https://www.npmjs.com/package/@roxyapi/ui-react) to render |
| Content | MDX files, no CMS |
| Tests | Vitest |
| Hosting | Anything that runs Next.js. Vercel in one click |

```bash
npm run dev        # develop
npm run build      # production build
npm test           # the drift guards: config, palettes, structured data, error mapping
npm run typecheck  # types
```
